## 3. Главная научная цель текущего этапа

На текущем корпусе нужно строго проверить следующую гипотезу:

Компактный и интерпретируемый набор морфометрических, цветовых и текстурных признаков, извлеченных из объектов detection-разметки и сегментационных масок, способен обеспечить устойчивое различение `blast / non-blast` на данных разных пациентов и дать практически полезный уровень image-level агрегирования для задач автоматического анализатора.

## 4. Ограничения текущих данных

Перед реализацией кода нужно принять следующие ограничения как часть постановки, а не как технические мелочи:

1. В разметке отсутствует разделение `lymphoblast` и `myeloblast`; следовательно, текущий эксперимент не может служить доказательством для подтипизации бластов.
2. Основной надежный уровень оценки — пациент, а не объект и не изображение.
3. В разметке присутствуют классы качества и артефактов (`Bad cells`, `artifacts`, `gumpricht shadows`), которые нельзя механически смешивать с морфологически корректными клетками без отдельного протокола.
4. Маски покрывают почти весь корпус, но не полностью; нужна двухветочная логика `mask-aware` и `bbox-only fallback`.
5. Часть label-файлов содержит class id в формате `float`, следовательно, парсер должен явно использовать преобразование `int(float(raw_class_id))`.
6. В сыром каталоге присутствует специальная директория `part-1`, не соответствующая формату patient id; до уточнения ее происхождения она должна быть исключена из основного patient-level протокола и рассматриваться только как возможный вспомогательный корпус.

### A1. Формирование аналитических подмножеств

#### Цель

Развести разные исследовательские режимы, чтобы не смешивать клинически осмысленную задачу распознавания клеток и задачу устойчивости к плохому качеству и артефактам.

#### Подмножества, которые нужно сформировать

1. `full_realistic_set`
   Используются все объекты, включая `Bad cells`, `artifacts` и `gumpricht shadows`.

2. `clean_cell_set`
   Исключаются только `Bad cells` и `artifacts`.
   Объем по текущей оценке для основного patient-level корпуса: `78 517` объектов.

3. `strict_morphology_set`
   Исключаются `Bad cells`, `artifacts` и `gumpricht shadows`.
   Объем по текущей оценке для основного patient-level корпуса: `74 917` объектов.

4. `mask_ready_set`
   Объекты, у которых есть валидная маска.

#### Почему это важно

- `full_realistic_set` нужен для анализаторно-ориентированной постановки.
- `clean_cell_set` нужен для более честной оценки распознавания клеток.
- `strict_morphology_set` нужен для сравнения текстурных и морфометрических признаков без сильного влияния шумовых классов.

### A2. Построение patient-level разбиений

#### Цель

Исключить утечку информации между выборками и подготовить единый протокол оценки для всех моделей.

#### Основные правила

1. Один пациент может входить только в один из наборов `train`, `validation`, `test`.
2. Все модели должны использовать одни и те же финальные разбиения.
3. Внутренняя настройка гиперпараметров должна происходить только внутри train-части.
4. В базовом протоколе участвуют только `122` patient-идентифицированных каталога; `part-1` в это разбиение не входит.

#### Рекомендуемый протокол

1. Финальный внешний hold-out по пациентам:
   - `70%` пациентов — train;
   - `15%` пациентов — validation;
   - `15%` пациентов — test.

2. Внутри train:
   - `StratifiedGroupKFold` или собственная стратификация по доле `blast` у пациента;
   - минимум `5` повторов для признаковых моделей.

#### Дополнительные таблицы для стратификации

Нужно посчитать для каждого пациента:

- число объектов;
- число `blast`;
- долю `blast`;
- число изображений;
- число изображений с `blast`;
- долю объектов без масок.

#### Выходные артефакты

- `outputs/splits/patient_summary.csv`
- `outputs/splits/split_v1.json`
- `outputs/splits/split_v1_train_patients.txt`
- `outputs/splits/split_v1_val_patients.txt`
- `outputs/splits/split_v1_test_patients.txt`
- `outputs/tables/class_distribution_by_split.csv`

#### Критерий завершения этапа

Разбиение стабильно, воспроизводимо и используется далее без ручных корректировок.

#### Проверяемые результаты этапа

- сформирован единый `split_v1`, используемый всеми последующими моделями;
- для каждого пациента определена принадлежность к `train`, `validation` или `test`;
- рассчитана таблица стратификации по числу объектов, числу `blast`, доле `blast` и числу изображений;
- сохранены текстовые списки пациентов и JSON-описание разбиения.

#### Как оценить качество результата

- пересечение множеств пациентов между `train`, `validation` и `test` равно `0`;
- распределение класса `blast` и размера выборки между частями разбиения не содержит грубых перекосов;
- при повторной сборке с тем же `random_state` получается идентичный `split_v1`;
- `part-1` отсутствует во всех split-файлах;
- таблица `class_distribution_by_split.csv` позволяет отдельно проверить баланс по объектам и по пациентам.

#### Предлагаемые модули кода

- `tests/test_group_split_no_leakage.py`

### A4. Базовая объектная постановка `blast / non-blast`

#### Цель

Получить строгий и воспроизводимый baseline, который напрямую отвечает на главный вопрос статьи: полезны ли текстурные признаки в условиях реальных данных и patient-level оценки.

#### Основная целевая переменная

- `target_binary = 1`, если `class_id == 7`;
- `target_binary = 0` для остальных объектов выбранного подмножества.

#### Какие версии задачи нужно решить

1. `Task A4.1`
   `blast / non-blast` на `clean_cell_set`.

2. `Task A4.2`
   `blast / non-blast` на `strict_morphology_set`.

3. `Task A4.3`
   `blast / everything_else` на `full_realistic_set`.

Последняя постановка особенно важна для автоматического анализатора, потому что она показывает устойчивость модели к артефактам и классам плохого качества.

#### Артефакты

- `outputs/dataset_index/task_a41_clean_binary.parquet`
- `outputs/dataset_index/task_a42_strict_binary.parquet`
- `outputs/dataset_index/task_a43_full_realistic_binary.parquet`
- `outputs/tables/task_prevalence_summary.csv`
- `outputs/reports/task_definition_report.md`

#### Проверяемые результаты этапа

- подготовлены три бинарные постановки с явно определенной целевой переменной;
- для каждой постановки рассчитаны числа `blast`, `non-blast`, доля положительного класса и список hard-negative классов;
- зафиксировано, какие именно задачи используются для дальнейшего сравнения моделей.

#### Как оценить качество результата

- во всех трех файлах задача `blast / non-blast` формируется строго по правилу `class_id == 7`;
- `task_prevalence_summary.csv` позволяет проверить дисбаланс классов и сложность каждой постановки;
- во всех постановках сохраняется patient-level совместимость со `split_v1`;
- по отчету `task_definition_report.md` можно проверить, какие классы формируют трудные отрицательные примеры и насколько различаются режимы `clean`, `strict` и `full_realistic`.

### A5. Вычисление признаков

#### Семейства признаков, которые нужно реализовать

1. Морфометрические признаки:
   - площадь bbox;
   - площадь маски;
   - периметр маски;
   - круглотность;
   - эксцентриситет;
   - отношение площади маски к площади bbox;

   - вейвлет-признаки;
   - спектральные признаки Фурье при необходимости.

1. по `bbox_crop`;

### A6. Признаковые baseline-модели

#### Цель

Сравнить информативность разных наборов признаков до подключения сложных нейросетевых архитектур.

#### Последовательность моделей

1. Logistic Regression
2. Linear SVM
3. RBF SVM
4. Random Forest
5. XGBoost или LightGBM

#### Для каждой модели нужно сравнить

1. только морфометрию;
2. только цвет;
3. только текстуру;
4. морфометрия + цвет;
5. морфометрия + текстура;
6. цвет + текстура;
7. полный набор признаков.

#### Метрики

Основные:

- PR-AUC;
- ROC-AUC;
- balanced accuracy;
- macro-F1;
- recall для класса `blast`;
- specificity;
- Matthews correlation coefficient.

Дополнительные:

- calibration error;
- Brier score;
- доверительные интервалы по повторам разбиения.

#### Артефакты

- `outputs/tables/object_level_metrics_baselines.csv`
- `outputs/tables/feature_group_ablation.csv`
- `outputs/predictions/object_level_baseline_predictions.parquet`
- `outputs/figures/pr_curves_baselines.png`
- `outputs/figures/roc_curves_baselines.png`
- `outputs/tables/baseline_model_leaderboard.csv`
- `outputs/figures/confusion_matrix_best_baseline.png`

#### Проверяемые результаты этапа

- обучены все запланированные признаковые baseline-модели на одинаковом `split_v1`;
- получена таблица метрик по каждой модели и каждому режиму признаков;
- построен лидерборд лучших моделей и лучший baseline для дальнейших абляций;
- сохранены object-level вероятности и классовые предсказания для последующего image-level агрегирования.

#### Как оценить качество результата

- в `object_level_metrics_baselines.csv` присутствуют все основные метрики и доверительные интервалы;
- `baseline_model_leaderboard.csv` позволяет проверить, какая комбинация `модель + режим признаков + subset` действительно лучшая;
- по confusion matrix и PR/ROC-кривым можно проверить не только среднее качество, но и характер ошибок;
- отдельно оценивается recall класса `blast`, так как именно он является клинически критичным.

### A7. Отбор признаков и интерпретация

#### Цель

#### Что нужно проверить

1. Устойчивость ranking признаков между разными фолдами.
2. Согласованность ranking между моделями.
3. Какие именно признаки стабильно усиливают разделение `blast / non-blast`.

#### Рекомендуемые методы

- permutation importance;
- SHAP для деревьев и бустинга;
- регуляризованные линейные модели;
- повторяемый отбор признаков с подсчетом частоты выбора.

#### Артефакты

- `outputs/tables/top_features_stability.csv`
- `outputs/figures/shap_summary_top_model.png`
- `outputs/figures/feature_importance_stability.png`
- `reports/feature_interpretation_notes.md`

#### Проверяемые результаты этапа

- получен устойчивый ranking наиболее информативных признаков;
- зафиксирована повторяемость top-признаков между фолдами и между моделями;
- для лучшей модели построены интерпретационные диаграммы и текстовые пояснения.

#### Как оценить качество результата

- `top_features_stability.csv` позволяет проверить частоту появления признака в top-N списках разных запусков;

- в `feature_interpretation_notes.md` должно быть указано, какие признаки имеют ясный морфологический смысл и какие остаются только статистически значимыми;
- результат считается сильным, если ключевые признаки не меняются хаотично от фолда к фолду.

#### Практический результат

Этот этап должен дать содержательные формулировки для статьи и диссертации: какие текстурные и морфометрические свойства чаще всего ассоциированы с бластными объектами в данном корпусе.

### A10. Анализаторно-ориентированное image-level агрегирование

#### Цель

Сделать переход от классификации отдельного объекта к постановке, более близкой к автоматическому анализатору: содержит ли изображение хотя бы один вероятный бласт и насколько высоко модель ранжирует такие изображения.

#### Доступные сейчас постановки

1. `image_contains_blast`
   Изображение считается положительным, если среди его объектов есть хотя бы один `blast`.

2. `top_k_image_ranking`
   Изображения сортируются по максимальному или агрегированному blast-score объектов.

3. `patient_risk_score`
   Для пациента агрегируется статистика по всем изображениям:
   - максимум blast-score;
   - доля изображений с blast-score выше порога;
   - средний score по top-K объектам.

#### Способы агрегирования, которые нужно сравнить

1. `max object score`;
2. `mean top-3 scores`;
3. `count of objects above threshold`;
4. `soft noisy-or`.

#### Метрики

- image-level PR-AUC;
- image-level recall при фиксированной доле ложных тревог;
- patient-level sensitivity;
- top-K recall для изображений с бластами.

#### Почему это важно

Этот этап лучше всего связывает текущие данные со статьей и будущей диссертацией: даже не имея настоящих slide-level меток, мы уже можем проверить, насколько object-level модель пригодна как подсистема отбора подозрительных изображений для анализатора.

#### Артефакты

- `outputs/tables/image_level_metrics.csv`
- `outputs/tables/patient_level_ranking_metrics.csv`
- `outputs/predictions/image_level_scores.parquet`
- `outputs/predictions/patient_level_scores.parquet`
- `outputs/figures/image_ranking_recall_curve.png`
- `outputs/figures/patient_ranking_curve.png`

#### Проверяемые результаты этапа

- object-level модель переведена в image-level и patient-level постановку;
- рассчитаны image-level и patient-level score для всех объектов наблюдения;
- определен лучший способ агрегирования object scores для анализаторно-ориентированного сценария.

#### Как оценить качество результата

- `image_level_metrics.csv` позволяет сравнить разные способы агрегирования по PR-AUC и recall при ограниченной ложной тревоге;
- `patient_level_ranking_metrics.csv` позволяет оценить, насколько хорошо пациент с бластами поднимается в верх ранжированного списка;
- `image_ranking_recall_curve.png` и `patient_ranking_curve.png` позволяют визуально проверить пригодность модели для предварительного отбора подозрительных изображений и пациентов;
- этап считается особенно успешным, если даже простой object-level baseline обеспечивает высокое top-K покрытие изображений и пациентов с бластами.

## 7. Часть B. Эксперименты, требующие дополнительных данных

### B1. Разделение лимфобластов и миелобластов

Для этой постановки обязательно нужен новый корпус или доразметка текущего корпуса, где класс `blast` будет разделен минимум на:

- `lymphoblast`;
- `myeloblast`;
- при необходимости `ambiguous blast-like`.

Только после этого можно переносить на практику исходную гипотезу статьи о class-specific текстурных различиях лимфобластов и миелобластов.

### B2. Полноценный эксперимент по доменному сдвигу

Нужны данные хотя бы из еще одного источника:

- другая лаборатория;
- другой сканер;
- другая методика окраски;
- другой режим подготовки препаратов.

Тогда можно сравнивать:

- handcrafted only;
- deep only;
- hybrid.

И только в этом случае будет корректно обсуждать устойчивость к междоменному переносу.

### B3. Slide-level и case-level постановка

Чтобы реализовать сильную анализаторно-ориентированную схему, требуются:

- полноценные slide-level метки;
- case-level клинические метки;
- желательно информация о конечном диагнозе или наличии резидуальных бластов на уровне случая.

Без этого текущий этап может дать только image-level приближение.

### B4. Экспертная клиническая интерпретация

Для глубоких моделей и усиленной интерпретации нужны:

- ручные контуры ядра и цитоплазмы;
- выделение нуклеол;
- при AML — специальные признаки типа палочек Ауэра;
- экспертная blinded-оценка областей, на которые опирается модель.

### B5. Продвинутые deep и MIL-схемы

Полноценные transformer-подобные модели, MIL, weak supervision и причинно-структурные архитектуры имеет смысл выносить в отдельный этап только после того, как будет завершен и воспроизводимо оформлен этап A.

## 8. Приоритеты реализации

Код следует писать именно в таком порядке:

1. `A0` — индексация и аудит.
2. `A1` — аналитические подмножества.
3. `A2` — patient-level split.
4. `A3` — извлечение ROI.

Такой порядок выбран специально: сначала нужно получить строго воспроизводимый, интерпретируемый и диссертационно полезный baseline, а уже потом усложнять модельную часть.
